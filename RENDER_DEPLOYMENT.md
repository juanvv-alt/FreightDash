# Render Deployment Guide for FreightDash

## Prerequisites

- GitHub account with your FreightDash repository
- Render account (https://render.com)
- All changes committed to git

## 📋 Pre-Deployment Checklist

Before deploying to Render, ensure:

- [ ] All files are committed to git: `git add -A && git commit`
- [ ] `.env` file is **not** committed (sensitive data)
- [ ] `.env.example` contains only non-sensitive template vars
- [ ] `render.yaml` is properly configured
- [ ] Dockerfile exists in repository root
- [ ] requirements.txt has all dependencies

## 🚀 Step-by-Step Deployment

### Step 1: Push to GitHub

```bash
cd /Users/juanvanvyve/Documents/workspaceVS/FreightDash
git push origin main
```

### Step 2: Create Render Account & Connect GitHub

1. Go to https://render.com
2. Sign up → Select "GitHub" 
3. Authorize Render to access your GitHub account
4. Select your `FreightDash` repository

### Step 3: Deploy Using render.yaml (Recommended)

When you connected your repo, Render should auto-detect `render.yaml`:

1. **Dashboard** → **New +** → **Web Service**
2. Select your `FreightDash` repository
3. Click **Deploy** (Render reads render.yaml automatically)

Render will:
- ✅ Read `render.yaml` configuration
- ✅ Build using Dockerfile
- ✅ Create PostgreSQL database
- ✅ Deploy to Singapore region
- ✅ Set environment variables

### Step 4: Configure Environment Variables (If Needed)

If Render doesn't auto-detect all environment variables:

1. **Service** → **Environment**
2. Add missing variables:
   ```
   DEBUG=False
   SECRET_KEY=<auto-generated>
   ENVIRONMENT=production
   TZ=Asia/Singapore
   ALLOWED_HOSTS=*.onrender.com
   ```

### Step 5: Run Initial Migrations

After deployment, run migrations:

```bash
# In Render Dashboard → Service → Shell
python manage.py migrate
python manage.py create_admin
```

Or use:
```bash
# From your local machine
curl https://api.render.com/deploys/YOUR_DEPLOY_ID/shell
```

## 🔐 Security Setup

### 1. Change Admin Password

Access the web shell in Render dashboard:
```bash
python manage.py changepassword admin
```

Or create new superuser:
```bash
python manage.py createsuperuser
```

### 2. Generate Strong SECRET_KEY

Render auto-generates this, but verify it's set:
1. **Service** → **Environment**
2. Check `SECRET_KEY` is long and random

### 3. Setup Custom Domain (Optional)

1. **Service** → **Settings** → **Custom Domain**
2. Enter your domain (e.g., `freight.yourdomain.com`)
3. Add CNAME record to your DNS provider

## 🐛 Troubleshooting

### Deploy Fails: "Dockerfile not found"

**Solution:**
- Ensure Dockerfile is in repository root (not in a subdirectory)
- Run: `git add -A && git commit && git push`
- Trigger redeploy from Render dashboard

### Deploy Fails: "requirements.txt not found"

**Solution:**
- Verify `requirements.txt` is in root directory
- List files: `ls -la` should show Dockerfile and requirements.txt

### Application crashes after deploy

**Check logs:**
1. **Dashboard** → **Logs** (top right)
2. Look for error messages
3. Common issues:
   - Missing DATABASE_URL env var
   - SECRET_KEY not set
   - Migrations not applied (run in shell)

### Admin page returns 500 error

**Solutions:**
1. Ensure `python manage.py migrate` was run
2. Check that admin user exists: `python manage.py create_admin`
3. Verify database connection in logs

## 📡 AIS Supply Forecast Worker

The `/supply-forecast/` feature needs a long-lived process consuming the
[aisstream.io](https://aisstream.io) websocket. `render.yaml` defines a
`freightdash-ais` **worker** service running
`python manage.py ingest_ais --with-aggregation`.

1. Get a free API key from aisstream.io.
2. Set `AISSTREAM_API_KEY` on the `freightdash-ais` service in the Render
   dashboard (it is declared `sync: false`, so it is not stored in git).
3. Run `python manage.py seed_pacific_ports` once (e.g. from a one-off shell)
   so the geofences exist.

The worker maintains vessel state, emits port-call events, and builds the daily
`DailySupplySnapshot` + `SupplySignal` shortly after local midnight. Signals read
*insufficient* until ~2 weeks of history accumulate, then switch to a z-score
heuristic and eventually a regression — coverage is shown in the page banner.

> ⚠️ One websocket connection per free key. Do not run the docker-compose
> `ais_ingester` and the Render worker on the same key simultaneously.

**Degraded path (no paid worker):** replace the worker with two Render cron jobs:

```yaml
  - type: cron
    name: freightdash-ais-poll
    schedule: "*/20 * * * *"
    dockerCommand: python manage.py ingest_ais --duration-seconds 1080
  - type: cron
    name: freightdash-supply-agg
    schedule: "15 0 * * *"
    dockerCommand: python manage.py aggregate_supply
```

Each poll ingests for 18 of every 20 minutes. Geofence transitions are still
detected on the next position inside/outside a fence, so port calls are delayed
rather than lost — though in/out flicker during the 2-minute gaps is missed.
